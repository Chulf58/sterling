# Grill-intent SOP (§7.6)

Run BEFORE planning, on the user's verbatim words only — no paraphrase, no pre-stuffing, no thoroughness directives. Capturing the user verbatim is a fidelity rule on **their** words — not a gag on yours: you are a reasoning partner here, not a scribe.

1. **One question at a time.** Ask the single most important unresolved question, with options and a recommendation. Wait. Then the next. Never batch Q1/Q2/Q3.
2. **Adaptive passes; the council is an optional one.** The interview proceeds in passes (opening intent questions, then follow-ups — floor of two). Once intent is thick — the readiness check: (a) the change in one sentence the user would endorse, (b) the problem/value it serves, (c) where it lands, sharply enough that a hat could aim a KB + code query — you MAY ask the user whether to run `/sterling:council` (skills/council/SKILL.md): an opt-in deliberation pass returning a multi-perspective question agenda + risks + alternatives. The user decides; never auto-fire; never on thin intent (close the gap with a targeted pass first). Drain the returned agenda one question at a time (step 1 holds); route its research questions through step 6. Re-plan passes adaptively until every brief section is resolved-or-waived — the interview must run fully with zero council.
3. **Drive every brief section to resolved-or-waived:** problem, feature, user_stated criteria/constraints, acceptance criteria (phrased as end-to-end observable behavior), out_of_scope. A waived section is recorded as waived — never silently empty.
4. **Concept inventory (decision `7208729b`):** before the brief closes, inventory the domain concepts the feature **introduces or changes** — concept FAMILIES, at the level where intent and interactions are designed (item category, entity type, core mechanic; API resource family, document type, integration surface); members live inside the family article, and a sub-concept splits out only when it accrues its own intent + interactions distinct from the parent. Query first (`knowledge_query` by family term / `concept_family`) — a family may already have its article. Every NEW family gets its concept article created at grill/brief time (`feature_article` with `concept_family`: what it IS + members, INTENT + INTERACTIONS cross-referenced by sibling slug, owning code files) and its UUID listed in the brief's `reconcile_list`; every CHANGED family's existing article is flagged for reconciliation there too. An empty inventory is recorded as waived, never silently skipped.
5. **Attribution discipline:** the user's statements go to `user_stated` verbatim-faithfully. Your inferences — and any materially-better approach you see (outcome-changing only, never taste) — go to `conductor_proposals`, each marked confirmed/unconfirmed by an explicit user response. An unanswered recommendation is NOT an accepted one. Same rule for **limitations**: any limitation the chosen approach would carry — especially one that weakens or defeats the feature's purpose (it "works" but can't do what it's *for*) — is surfaced as an explicit **keep-or-solve question** and recorded confirmed/waived in the brief. A limitation you foresaw but did not raise is NOT an accepted one.
6. **Research trigger:** "can we do X?" that the user cannot answer → dispatch the researcher mid-grill (bounded question, capped budget); never guess, never park it silently.
7. Output: brief sections written via `knowledge_create`/update on the draft brief; open proposals listed back to the user for explicit accept/reject.
8. **Shared ground — glossary at entry, hard stop on confusion (board ef157057, 2026-07-19).** An answer built on a term the user did not understand *looks* like a decision but is not an informed one (trigger, user verbatim: questions "built on internal jargon … my answers looked like decisions but weren't informed ones"). Two duties: (a) **At interview entry**, present a short plain-language glossary of the system terms the interview will lean on — seeded from `concept_family` articles where they exist (the step-4 inventory machinery), plainly worded where they don't; any NEW system term introduced mid-interview is defined in the question that introduces it. (b) **A "what is X?" answer is a HARD STOP** — halt the interview, rebuild shared ground in plain language, and re-validate every prior answer that leaned on the misunderstood term before asking anything new. Both duties are prose with no enforcement surface (no hook fires on assistant text) — which is exactly why skipping them under momentum is the recorded failure mode; don't.
