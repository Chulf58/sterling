# Planning SOP (§7.6)

Read-only discipline throughout (plan-mode-shaped — adjudicated; native plan
mode is not load-bearing, §16.2 step 8). Iterable steps; each output is a required brief section:

1. **Technical design.** Pre-dispatch the implementation-architect when complexity is visible at intake (the common case). Dispatch explorer/researcher for heavy reads. Small bounded questions: inline lookups (web + knowledge tools) are licensed — capture findings; big unknowns → researcher. Output: `technical_design` (approach, interfaces with contracts, shared structures).
2. **Decomposition into phases**, informed by the design. Re-read blast radius if the design widened scope. Each phase spec declares its `files` + `rank_terms` — prep's staging inputs are planning outputs, never improvised at staging time. Every phase's interfaces must be declared (the test-writer's spawn fails loud otherwise). Breadth rule: every phase stays within `config.difficulty.split_interface_threshold` (default 3) interfaces; over that ⇒ split the phase before the gate — the `phase_over_wide` flag reports it, and prep and H8 refuse to stage or dispatch an over-wide phase.
3. **AC formalization + flags.** ACs as observable end-to-end behavior, each with `verifiable_at`. Difficulty is the PLANNER'S call, made with cited reasons against the depth inputs (prep retrieval volume vs `thin_knowledge_retrieval_threshold`, algorithmic complexity, ambiguity residue, poorly-covered external integration) — the human is never asked to determine hardness (decision a48c74cf). DEFAULT TO NORMAL when the depth inputs don't clearly indicate hard; under-rating is detected mechanically by the loop caps (§5.1) and answered by upgrade or re-scope, never guessed up front. Bigness is NOT hardness: an over-wide phase SPLITS, it never upgrades. Risk flags `security_relevant`/`perf_sensitive` proposed here, human-confirmed at the gate.
4. **Brief assembly + decision capture.** Planning retrieval runs with `include_unconfirmed: true`; any `derived_unconfirmed` record relied on is confirmed-or-killed on the spot (§3.2.6). Design decisions → decision records, cited in `decisions_made`. **Carry the grill's concept inventory into `blast_radius.reconcile_list`** (decision `7208729b`): every concept article the grill created or flagged (grill-intent step 4) lands there by UUID — that field is what completeness-check and the disposal gate enforce, so an inventory that never reaches it has no teeth. If the design surfaced a NEW concept family the grill missed, create its concept article now (the read-only discipline binds the working tree, not the store) and list it too.

**Feature-sizing check:** a feature too big to plan in one conversation is split into features (P7). The brief's AC + scope sections lock at the gate — hooks will enforce them for hours. Re-scope, don't resume: a phase that proves over-scoped mid-run is re-scoped (split) and redone, never stopped and resumed — a resume compounds an already-large context (P7).
